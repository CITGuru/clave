# Clave ‚Äî Implementation Status

A snapshot of what's built, what's wired together, and what still needs platform-specific approvals or hosts. For what Clave *is* and how to run it, see the [README](README.md). For architecture and subsystem design, see [`docs/`](docs/README.md). For the tracked list of work buildable on any dev machine today (no Windows host / Apple entitlement / driver signing), see the [non-OS-gated backlog](BACKLOG.md).

## Phases

- **Phase 1 (portable core) ‚Äî complete.** Policy brain, IPC contracts, and the daemon skeleton, all tested on any machine with no driver / entitlements / signing.
- **Phase 2 (network split-tunnel) ‚Äî in progress.** Shared routing + the **boringtun WireGuard data plane** (done, behind the `wireguard` feature: in-memory handshake + encrypted round-trip tested). Remaining: the real OS enforcement (Swift Network Extension; Windows WFP), which needs entitlements / a Windows host ‚Äî see per-crate READMEs and [doc 12](docs/12-signing-distribution-deployment.md).
- **Phase 3 (encrypted Clave Disk) ‚Äî crypto core complete; macOS mount real and Secure-Enclave-sealed.** The portable encrypting layer in `clave-volume` (**AES-256-XTS**, **KEK‚ÜíDEK** hierarchy with AES-KW wrapping, hardware-key-store and backing-container seams, **crypto-shred remote wipe**) is fail-closed and tested with no OS involved. The daemon owns this volume as a subsystem exactly like `SplitRouter`. On macOS, `clave-mac`'s `MacVolumeMount` (`src/volume.rs`) is a real mount: an `hdiutil`-created `AES-256` `SPARSEBUNDLE`, crypto-shred wipe on request. The passphrase is Secure-Enclave-sealed (`src/se_seal.rs`): a non-extractable P-256 key generated inside the SE (SE has no X25519 support, so this is a parallel construction to the X25519 enrollment sealed-box below), sealed/opened via ECDH + AES-KW, with the unwrap itself running inside the chip. Reaching the SE needs a properly signed, provisioned binary (`keychain-access-groups` is a restricted entitlement that SIP-disabled does *not* relax) ‚Äî `crates/clave-daemon-host` is the `#[no_mangle]` FFI shim (kept out of `clave-daemon`, which forbids unsafe code) that the real, signed `crates/clave-mac/macos/ClaveDaemonHost` app links. The daemon runs under one of two **profiles** that own separate disks: `SignedHost` (the app; SE-sealed, and it refuses to provision a software-only disk) and `Dev` (`cargo run`; unsigned, so its own throwaway disk with a plain-Keychain passphrase). Custody is fixed at container creation and never re-provisioned on open, so a sealed disk fails closed rather than being silently re-keyed. Not yet done: ES `AUTH_OPEN`-gating the mount (needs the same ES entitlement as process supervision ‚Äî see blockers below), and the Windows side (WinFsp mount, TPM key store, minifilter) ‚Äî see [doc 04](docs/04-encrypted-volume.md) and [doc 13](docs/13-build-roadmap.md).
- **Phase 5 (DLP surface) ‚Äî clipboard and screen-capture started (macOS).** Both are **detect-and-audit, not hard controls**, and neither can ever reach `Enforced` on macOS ‚Äî the platform simply does not offer the primitive. They are honest about that in code, in the enforcement report, and here.
  - **Clipboard** (`clave-mac/src/clipboard.rs`): polls the general pasteboard, tags a payload copied by a work app, and clears it when a personal app comes to the front and policy denies the transfer. Every observed work‚Üípersonal transfer is decided by `clave-core`'s `clip_decision` (per-format, from the tenant policy) and audited through `Daemon::decide_action`. macOS exposes no paste-interception API, so a paste timed inside the poll window still wins: the leak window is narrowed, the transfer is always recorded, and it is never claimed to be blocked (doc 05 ¬ß3.3).
  - **Screen capture** (`clave-mac/src/screen.rs`): notices capture tooling running while work windows are on screen (CGWindowList), puts it through the new `ScreenPolicy` in `clave-core` (`Action::ScreenCapture` ‚Üí deny by default, with a sanctioned-capturer allow-list), and audits it as `ScreenCaptureOverWork`. macOS **cannot** exclude a window it does not own from capture ‚Äî `sharingType` is an instance property on your own `NSWindow` and injection is ruled out by SIP ‚Äî so `ScreenGuard::protect_window` now honestly returns `Unsupported` instead of reporting a success it cannot deliver (doc 07 ¬ß3.2). The one hard block macOS allows, ES `AUTH_EXEC`-denying `screencapture`, needs the Endpoint Security entitlement.
  - Both state machines are pure and unit-tested, and both are exercised against the real OS: `cargo run -p clave-mac --example clipboard_live` copies a secret in a work app, switches to a personal one, and asserts the system clipboard is empty; `--example screen_live` takes an actual screenshot and asserts it is reported over work content, once, and ignored when no work window is visible.
  - **Input taps** (`clave-mac/src/input.rs`): enumerates keyboard event taps (`CGGetEventTapList`) and, when a non-work process is tapping while a work app is focused, decides via the new `InputPolicy` (`Action::InputTap`) and audits it as `InputTapOverWork`. macOS ships no shippable kernel input filter (doc 06 ¬ß3.1), so this is visibility only; TCC's Input Monitoring prompt is the platform's real backstop. Verified live: `--example input_live` lists the real tappers on the machine.
  - **Overlay:** the Clave Edge border (`clave-mac/src/edge.rs`) was already drawn and running via a `CGWindowList` poll; its enforcement posture is now honestly `DevelopmentOnly` rather than `Unavailable` ‚Äî a UI affordance, never a control (doc 09 ¬ß3.3).
  - Remaining in Phase 5: the Windows side ‚Äî where the clipboard broker, `SetWindowDisplayAffinity`, and the keyboard-filter driver are genuinely *hard* controls (docs 05 ¬ß2, 06 ¬ß2, 07 ¬ß2).
- **Gateway control plane (`clave-proto`) ‚Äî complete, and now closed end-to-end.** Remote **wipe**, remote **lock**, and **policy updates** are authenticated: a detached **Ed25519** signature over a canonical envelope, verified against the **pinned** tenant key, with **monotonic-counter anti-replay** and a freshness window (doc 04 ¬ß6, doc 10 ¬ß2). The daemon's `apply_gateway_command` is the only path that can change device posture; forged or replayed commands change nothing. Outbound, the device-signed, **hash-chained audit spool** drains tamper-evidently to the gateway. The gateway now **ingests and verifies** that spool: `clave-gateway`'s `AuditLedger` (`src/audit_ingest.rs`) holds each enrolled device's verified chain position `(next_seq, head)` and admits a batch only if it continues that chain under the key the device enrolled with ‚Äî so a dropped batch surfaces as a sequence gap, a rewritten event as a hash mismatch, a forged one as a bad signature (doc 10 ¬ß6). A cross-crate end-to-end test (`clave-daemon/tests/gateway_end_to_end.rs`) proves the whole loop against a real `Gateway` and a real `Daemon`: a device enrolls ‚Üí the gateway commands a wipe ‚Üí the daemon verifies and obeys it ‚Üí the daemon's signed audit drains back ‚Üí the gateway verifies it, and rejects both a tampered and an impostor-signed report.

## How it fits together

- **Data plane & IPC.** The split-tunnel data plane is wired into the daemon's flow path (`SplitRouter`), and an authenticated Unix-socket IPC transport carries the shim‚Üîdaemon contracts.
- **Encrypted Clave Disk.** The crypto core (`clave-volume`) is driven by the daemon ‚Äî encrypting at rest, gating non-supervised callers, and crypto-shredding on wipe.
- **Gateway sync.** Remote wipe/lock/policy commands are Ed25519-verified and replay-protected, with a device-signed, hash-chained audit spool. A portable `GatewaySync` loop drives the pull‚Üíapply‚Üídrain‚Üíship exchange over a `GatewayLink` seam, with a real **mutual-TLS** link (`clave-proto`'s `mtls` feature: a rustls connector/acceptor ‚Äî each side presents a cert and verifies the peer against pinned roots ‚Äî producing the `TlsStream` the framed `transport` pump runs over). The gateway sync loop also has a **live framed transport** (`clave-proto`'s `transport` feature) that mTLS wraps in production.
- **Honest enforcement posture.** Every OS capability reports an `EnforcementStatus` ‚Äî `Enforced` vs a `DevelopmentOnly` stand-in vs `Unavailable` ‚Äî so a production build can't silently ship a dev-only fallback (doc 14 ¬ß5.3).
- **Platform adapters.** The macOS adapter (`clave-mac`) is a real `MacPlatform` reporting that honest posture ‚Äî the ES-fed supervisor, the shared split-tunnel classifier, the Secure-Enclave-sealed Clave Disk mount, the clipboard guard, and the screen-capture watch are wired (all `DevelopmentOnly`: supervision/network pending ES entitlement approval, the volume pending its ES `AUTH_OPEN` gate, and clipboard/screen permanently, since macOS can neither intercept a paste nor exclude a third-party window from capture); the overlay and input-tap watch report `DevelopmentOnly` (both are detect/draw/audit ‚Äî real, but never a control). The Windows adapter (`clave-win`) is the same `WindowsPlatform` shape, and the daemon binary links the right one per target.
- **Exec & path classification.** `classify_exec` matches a binary's **code-signature** against the signed app allow-list (doc 02 ¬ß3.2), so only vetted apps (or children of supervised processes) join the work zone, and each rule carries a `LaunchProfile` that resolves the app's HOME/temp **inside** the encrypted Clave Disk (doc 03 ¬ß6, doc 04 ¬ß5). `classify_path` rounds out the portable classifier trio (flow / exec / path): it tells the FS-redirection hook and the minifilter/ES gate whether a path is work-data, system-COW, or pass-through (doc 03 ¬ß3.3). A **learn mode** synthesizes candidate profiles from an app's observed footprint (doc 03 ¬ß6), and the **`clave-cli`** diagnostics binary surfaces the enforcement posture and dry-runs the classifiers.
- **Property & fault tests.** **`proptest`** tests pin the security-critical invariants (the `decide`/classifier logic, XTS + AES-KW round-trips, the audit chain, panic-free verification of untrusted bytes) across all inputs (doc 11 ¬ß6), and a fault-injection test confirms the volume is fail-closed under a mid-write "pull-the-plug" (doc 04 ¬ß7).
- **Clave launcher.** `Daemon::launchable_apps` lists the allow-listed work apps and `prepare_launch` resolves each one's contained spawn spec ‚Äî executable + env pointing into the encrypted disk (doc 00 ¬ß5.2). The **Tauri desktop app** (`apps/clave-launcher/`, Rust + React/Tailwind/shadcn) reaches these over the authenticated `clave-ipc` link (`LauncherClient` ‚Üî the daemon's `handle_launcher_request`, doc 10 ¬ß3), falling back to a demo policy when no daemon is running; the OS spawn+inject is the remaining layer. `clave-cli apps`/`launch` drive the same core from the terminal.
- **Enrollment & control plane.** The control-plane **gateway** (`clave-identity` + `clave-gateway`, doc 15) rounds out enrollment: console login, device-enrollment handshake, **device registration**, and the two doc 15 ¬ß9 step 5 enrollment artifacts a device receives on approval (the shared wire contract is `clave-proto`'s `EnrollmentGrant`) ‚Äî its **tenant-signed initial policy bundle** (`PolicyIssuer` ‚Üí a `SignedCommand` its pinned-key `GatewayVerifier` accepts) and its **wrapped volume key** (`VolumeKeyService` ‚Üí the Clave Disk DEK delivered to the device, either AES-KW-wrapped to a symmetric KEK for the dev bootstrap or, in production, **sealed to the device's X25519 hardware public key** via an ECIES sealed-box (`clave-volume`'s `seal_dek`, so the gateway holds nothing that can open it), doc 04 ¬ß2). The device side closes the loop: **`clave-daemon`'s enrollment client** (`DeviceEnrollment::accept`) pins the tenant key, verifies the policy through it, and opens the volume key with its hardware key ‚Äî producing exactly the material `Daemon::new` is built from. All over an Axum edge with sealed-cookie sessions ‚Äî portable over in-memory seams, with real **Postgres** + **WorkOS** adapters (`--features server`) exercised against a live database.
- **Quality gate.** **330 tests** pass; `cargo clippy --all-targets` is clean (one pre-existing warning in `clave-mac/src/keychain.rs`).

## What needs more than this Mac

The WireGuard data plane is **done** (boringtun, behind the `wireguard` feature). The macOS Xcode
project (`crates/clave-mac/macos/`) is real and builds today, but two of its three targets are
blocked on account tier, not on missing code:

| Next step | Needs |
|-----------|-------|
| ES System Extension activation (`ClaveES` target) | The **paid** Apple Developer Program ‚Äî confirmed directly from the portal: free personal teams cannot provision System Extension capability at all |
| Endpoint Security entitlement itself, for a shippable (non-SIP-disabled) posture | A **separate** Apple approval request after enrolling ‚Äî restricted entitlement, weeks-long manual review, doesn't come free with payment |
| Network Extension / split-tunnel (not yet started) | Likely the same System Extension capability block above ‚Äî untested, confirm once enrolled |
| Windows enforcement (`clave-win` `cfg(windows)`) | A Windows checkout; later EV-cert + driver signing ‚Äî [doc 12 ¬ß1](docs/12-signing-distribution-deployment.md) |
| Windows Clave Disk mount + hardware keys (`clave-win` `volume.rs`) | WinFsp mount over the `clave-volume` core + TPM key store; Windows minifilter (attestation-signed) ‚Äî macOS's mount is done, see above |

`ClaveDaemonHost` (the daemon's signed launch path, Secure-Enclave sealing) is **not** blocked ‚Äî
it's provisionable and working on the free tier today. Full signing state, exact error messages,
and the action list for when the paid account lands: `crates/clave-mac/macos/XCODE_NOTES.local.md`
(gitignored, account-specific ‚Äî regenerate locally, not in this checkout).

See the [build roadmap](docs/13-build-roadmap.md) for full sequencing.
